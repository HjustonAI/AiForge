# FORGE v2 — Implementation Start

This document converts the v2 architecture vision into a concrete kickoff plan with deliverables, implementation notes, and acceptance criteria.

---

## 1) Knowledge Management System (KMS) Framework

**Scope**
- Define ingestion → storage → retrieval flow for prompt knowledge.
- Align ingestion strategy with Karpathy-style source-first knowledge capture.

**Initial design**
- **Ingestion layers**
  - `raw`: source artifacts (research docs, transcripts, notes, prompt outcomes)
  - `distilled`: normalized insights (patterns, anti-patterns, constraints)
  - `compiled`: target-ready context bundles for Compose
- **Storage model**
  - Keep source files in `forge/seeds/` and project-level research folders.
  - Keep distilled operational knowledge in `forge/contexts/targets/*.ctx.md`.
  - Keep reusable outputs in `forge/arsenal/prompts/*.md`.
- **Retrieval model**
  - Primary keys: `target`, `mode`, `domain`, `quality`, `freshness`.
  - Secondary tags: `task_type`, `output_format`, `risk_level`.

**Acceptance criteria**
- KMS layer definitions documented and approved.
- Tag taxonomy published and used consistently in new v2 assets.

---

## 2) Compose Mode Specifications

**Purpose**
- Compose generates:
  1. final prompts,
  2. operational guidelines,
  3. reusable meta prompts.

**Input contract**
- `intent`, `target_model`, optional `domain`, optional `constraints`, optional `reference_assets`.

**Output contract**
- `prompt.md` (or inline prompt)
- `meta.md` (reasoning blueprint + reusable pattern)
- `trace.json` (selected sources/tags and confidence)

**Supported use-case families**
- Personal tracking (journaling, habit systems, learning loops)
- Research workflows (exploration, synthesis, citation-aware prompts)
- Business workflows (marketing, sales ops, product discovery)

**Acceptance criteria**
- Compose I/O contract documented.
- At least one example per use-case family defined.

---

## 3) Distillation Process

**Pipeline**
1. Source intake and quality scoring.
2. Extraction pass (facts, leverage points, failure modes, examples).
3. Normalization pass (template mapping into `.ctx.md` sections).
4. Tagging pass (KMS taxonomy).
5. Validation pass (coverage, token budget, freshness metadata).

**Tagging standard**
- Required tags: `target`, `mode`, `domain`, `quality_score`, `last_validated`.
- Optional tags: `style`, `industry`, `audience`, `compliance`.

**Acceptance criteria**
- Distillation checklist added to v2 process docs.
- Every new distilled asset includes required tags.

---

## 4) Deploy Mode Architecture

**Role**
- Deploy is a Compose derivative that executes prompts against external systems and captures outputs.

**Workflow**
1. Resolve Compose artifact set.
2. Validate execution requirements (tooling, permissions, paths).
3. Execute against selected runner/toolchain.
4. Persist outputs + run metadata.
5. Feed quality signals back into KMS/arsenal.

**Acceptance criteria**
- Deploy workflow documented with pre/post conditions.
- Output capture format agreed (`result.md` + `run.json` minimum).

---

## 5) Meta Prompt Logging System

**Goal**
- Log every reusable meta prompt generated by Compose with searchable metadata.

**Proposed index schema**
- `id`, `created_at`, `target_model`, `mode`, `tags`, `quality_rating`, `source_refs`, `path`.

**Storage**
- Canonical entries in `forge/arsenal/_index.md`.
- Meta prompt bodies in `forge/arsenal/prompts/`.

**Acceptance criteria**
- Schema finalized.
- New meta prompts added to index via a standard template.

---

## 6) Inter-Agent Communication Protocols

**Baseline protocol**
- Prefer structured messages with fields: `agent`, `task`, `status`, `artifacts`, `handoff`.

**GitHub Gist channel rules**
- Use gists only for non-sensitive coordination payloads.
- Include reference IDs back to local artifacts.
- Never place secrets or proprietary data in gist payloads.

**Acceptance criteria**
- Message schema defined and documented.
- Security constraints for gist usage explicitly documented.

## 7) User Interface / Experience Plan

**Initial UX scope**
- Mode-first command entry (Compose / Distill / Deploy).
- Asset browser (contexts, arsenal, seeds, runs).
- Retrieval inspector (what was selected and why).

**Feedback plan**
- Lightweight interviews with target users.
- Collect top friction points and map to UI backlog.

**Acceptance criteria**
- MVP UX flows documented.
- First feedback session planned with review prompts.

---

## 8) Token Efficiency Strategy

**Principles**
- Retrieve less, retrieve better.
- Prefer distilled summaries over raw long context.
- Cache compiled artifacts by target + intent class.

**Operational controls**
- Token budgets by mode (Compose / Distill / Deploy).
- Tiered retrieval: core context first, optional expansions on demand.
- Automatic stale-context warnings to avoid redundant tokens.

**Acceptance criteria**
- Budget table defined for each mode.
- Retrieval policy documented with escalation thresholds.

**Cadence**

**Workshop outputs**
- Decision log updates.
- Ranked backlog changes.
- Risk register updates.

**Acceptance criteria**
- Workshop templates prepared.
- First two sessions scheduled with owners.

## 10) Phased Roadmap

### Phase 0 — Foundation (Weeks 1–2)
- Finalize KMS schema and Compose/Distill/Deploy specs.
- Agree metadata/tag standards and logging format.

### Phase 1 — Core Build (Weeks 3–5)
- Implement Compose v2 contracts and meta prompt logging.
- Implement Distillation v2 tagging + validation workflow.

### Phase 2 — Deploy + Agent Protocols (Weeks 6–8)
- Implement Deploy execution flow and output capture.
- Introduce inter-agent protocol and gist channel governance.

### Phase 3 — UX + Optimization (Weeks 9–10)
- Deliver initial UI flows and retrieval inspector.
- Apply token efficiency strategy and measure savings.

### Phase 4 — Operationalization (Weeks 11–12)
- Run team workshops, fold feedback into backlog.
- Final v2 readiness review and go/no-go decision.

**Milestone ownership (initial)**
- Architecture Lead: KMS, mode contracts
- Platform Lead: Deploy, protocol, integrations
- Product/UX Lead: UI flows, user feedback
- Ops Lead: token economics, reporting cadence

**Exit criteria**
- All 10 tracks have documented specs, owners, and measurable acceptance checks.
